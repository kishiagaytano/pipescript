Pipe>>Script

Most data code isn't logic.
It's cleanup.

A minimal, typed language built around one idea — data should flow through the operations that clean it.

<Badilla | Gaytano | Lastrollo | Regalado> <TS31> <July>

The Problem

Every dataset shows up dirty.

Negatives that should be positive. Nulls and blanks. Values in the wrong type. Before a single insight, you spend your time cleaning.

And it's always the same handful of steps, in the same order — repetitive, sequential, boilerplate.

raw ages

[ 25, -3, null, 41, -9, null ]
✕ negatives ✕ nulls ✕ unsorted

The Friction

General-purpose tools are overkill for a five-step cleanup.

The intent

remove negatives  →  drop nulls  →  fill defaults  →  sort

A straight line. Four steps.

The ceremony

import pandas as pd
df = pd.DataFrame(...)
df = df[df.age >= 0]
df = df.dropna()
df = df.fillna(0)
df = df.sort_values(...).reset_index()

The task is a straight line. The code shouldn't zig-zag.

The Solution

PipeScript makes the straight line the syntax.

local Int[] clean = ages >> removeNegatives() >> fillNull(0) >> sort();

Read it left to right. Each stage's output becomes the next stage's input. The >> operator is the whole language's spine.

40+ built-in cleaning verbs no imports statically typed reads like the intent

Program Structure

Anatomy of a PipeScript program

global Int passingScore = 50;

class Grader {
  String evaluate(Int score) {
    if (score < passingScore) { return "FAIL"; }
    return "PASS";
  }
}

pipeline {
  local Int[] scores = [72, 40, 88];
  Grader g = new Grader();
  scores >> g.evaluate();
}
global
Program-wide constants & state.
class
Data + behaviour (Phase 8).
pipeline
The single execution block.
local
Block-scoped declarations.
>>
Feed a value through a stage.

Top-level order is free: globals, classes, functions, and one pipeline.

Under the Hood · Architecture

The interpreter is, itself, a pipeline.

source.ps text >> LexerPHASE 2 >> ParserPHASE 3–4 >> SemanticPHASE 5 >> InterpreterPHASE 6–8 >> result

One function — PipeScriptEngine().run() — chains every stage and never throws. Each phase either passes data forward, or reports a structured error.

every error → { phase, message, line } — so the editor can point at the exact spot.

Phase 2 · Lexical Analysis

Text becomes tokens.

ages >> removeNegatives()

IDENT ages PIPE >> IDENT removeNegatives LPAREN ( RPAREN )
  • One left-to-right scan of the raw text
  • Tracks line & column on every token
  • Handles // comments & string escapes
  • Greedily folds >> into a single PIPE token
  • 18 reserved words separated from identifiers

The scanner decides, character by character, where > ends and >> begins — the smallest, most literal reading of the program.

Phase 3 · Syntax Analysis

Tokens become a tree.

local Int[] cleaned = ages >> removeNegatives() >> fillNull(0);

VarDecl (local · Int · cleaned)
└─ PipelineExpr
   ├─ source: Identifier ages
   └─ steps:
      ├─ Call removeNegatives()
      └─ Call fillNull(0)

Recursive-descent grammar

program  → (global | class | func
            | pipeline)*
pipeline → 'pipeline' block
expr     → orExpr ('>>' step)*

>> binds looser than every arithmetic and logical operator — so a full expression can flow into a pipe.

Phase 4 · Names, Scope & Binding

Every name knows where it lives.

global
Int passingScore
pipeline
Int[] scores   Grader g
for-body
Int i
  • Scopes are chained frames: global → pipeline → block
  • Lookup walks outward until the name is found
  • Inner declarations shadow outer ones
  • global vs local set where a binding is born
  • Assigning an undeclared name is caught, not guessed

Phase 5 · Semantic Analysis

Catch the mistake before it runs.

  • Type mismatches on assignment
  • Undeclared & re-declared variables
  • Non-Bool if / while conditions
  • Wrong argument counts
  • Unknown class, method, or field
  • Illegal operators between types

A real caught error

local Int x = 5;
x = "hello";

Semantic · line 3
Cannot assign a 'String' value to 'x' (declared as 'Int').

Two passes: collect every definition first, then type-check the tree. If semantics fail, the interpreter never runs.

Phase 6 · Control Flow

Decisions and loops.

local Int total = 0;
for (local Int i = 0; i < len(scores); i = i + 1) {
  if (get(scores, i) > 50) {
    total = total + 1;
  }
}
if / else
Branch on any truthy value.
for
Classic init · cond · update.
while
Loop until the condition breaks.
return
Unwinds cleanly out of a call.

safety — a 100,000-iteration guard turns runaway loops into a clean error, never a hang.

truthiness: 0, "", [ ], null → false

Phase 7 · Data Types

Phase 8 · Object Orientation

Classes that flow through pipes.

class Grader {
  String evaluate(Int score) {
    if (score < passingScore) {
      return "FAIL";
    }
    return "PASS";
  }
}
// one method, mapped over a list:
scores >> g.evaluate();
  • Class fields + methods, created with new
  • Encapsulation — methods read their own fields
  • Field & method access via .

The payoff

A method dropped into a pipe auto-maps over every element:

→ [ "PASS", "FAIL", "PASS", "FAIL", "PASS" ]
